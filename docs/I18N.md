# i18n — Internationalization Guide

> **Status**: Foundation live. Core UI migrations pending.
> **Stack**: `next-intl` v4, cookie-based locale detection, domain-based defaults.
> **Principle**: modular — any locale added in minutes, zero code changes elsewhere.

## Currently supported locales

| Code | Language | Native | Default on domain | Status |
|---|---|---|---|---|
| `en` | English | English | modulca.eu | ✅ Full (source of truth) |
| `ro` | Romanian | Română | modulca.ro | ✅ Core UI translated (Footer done, rest pending migration) |
| `nl` | Dutch | Nederlands | modulca.nl | ✅ Core UI translated (Footer done, rest pending migration) |

## How locale is detected (priority order)

1. **User preference** — cookie `modulca-locale` (set when user clicks LanguageSwitcher)
2. **Domain** — `modulca.ro` → `ro`, `modulca.nl` → `nl`, else default
3. **Browser** — `Accept-Language` header first preferred match
4. **Default** — English

See `src/i18n/config.ts` → `detectLocale()`.

## Adding a new locale (e.g. Spanish "es")

Total time: ~15 min + translation time.

1. Edit `src/i18n/config.ts`:
   ```ts
   export const LOCALES = ["en", "ro", "nl", "es"] as const;

   export const LOCALE_LABELS = {
     ...,
     es: { native: "Español", english: "Spanish", flag: "🇪🇸" },
   };
   ```
2. Copy `src/i18n/messages/en.json` to `src/i18n/messages/es.json` and translate values.
3. (Optional) If you own a Spanish domain, add it to `DOMAIN_LOCALE_MAP`.
4. Done — LanguageSwitcher picks it up automatically; fallback to `en` for any missing keys.

## Adding new translation strings

**Rule: never hard-code user-facing text.** Use translation keys.

1. Pick a namespace path: e.g. `home.hero.title`, `nav.library`, `auth.errors.weakPassword`.
2. Add the key + value to `src/i18n/messages/en.json` (source of truth).
3. Mirror in `ro.json` and `nl.json` (missing keys fall back to EN).
4. Use in code:
   ```tsx
   // Server Component
   import { useTranslations } from "next-intl";
   const t = useTranslations("home.hero");
   return <h1>{t("title")}</h1>;

   // Client Component — same hook, auto-detected
   "use client";
   import { useTranslations } from "next-intl";
   // ... same pattern
   ```

## Migrating an existing component to i18n

Example: `MyComponent.tsx` with hardcoded English strings.

Before:
```tsx
return <button>Get started</button>;
```

After:
```tsx
"use client"; // if not already
import { useTranslations } from "next-intl";

export default function MyComponent() {
  const t = useTranslations("home");
  return <button>{t("hero.ctaPrimary")}</button>;
}
```

Steps:
1. Find all hard-coded user-facing strings.
2. For each, add translation key in `en.json` (and mirror in ro/nl).
3. Import `useTranslations` with the namespace.
4. Replace strings with `t("key")`.
5. Test: switch language and verify.

## Architecture notes

- **No route-based locales** (no `/ro/...` URLs). Clean URL = cleaner SEO + less refactor.
- **Cookie + domain + header** = covers 99% of cases without URL pollution.
- **Fallback chain** in next-intl: requested locale → default (en) → raw key. Missing translations never break UI.
- **Server components** can use `useTranslations` from `next-intl` (same API).
- **Client components** get translations via `NextIntlClientProvider` in root `layout.tsx`.

## Technical reference

- `src/i18n/config.ts` — locales list, detection logic, domain mapping
- `src/i18n/request.ts` — per-request locale + message loading (next-intl)
- `src/i18n/messages/*.json` — translation files (one per locale)
- `src/features/shared/components/LanguageSwitcher.tsx` — dropdown + inline variants
- `src/app/api/locale/route.ts` — POST to set cookie
- `next.config.js` — wraps config with `createNextIntlPlugin("./src/i18n/request.ts")`

## Testing locally

1. Start dev server: `npm run dev`
2. Open `http://localhost:3000` — default locale (EN)
3. Footer → inline LanguageSwitcher → click RO or NL
4. Page reloads with new locale; footer labels change
5. Inspect `document.cookie` — should have `modulca-locale=ro`
6. Clear cookie to reset

## Known limitations

- Pluralization not yet used (add via `t("count", { n })` when needed — ICU format supported).
- RTL languages (Arabic, Hebrew) not supported (would need `dir="rtl"` logic — trivial to add).
- Content translation (blog articles, KB) is MANUAL per-locale — not yet a content-translation pipeline.
